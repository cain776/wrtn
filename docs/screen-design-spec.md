## 1. 공통 UX 원칙

### 1.1 한 제품처럼 보이게 하는 구조

- 상단 바에는 테넌트(`㈜한빛리테일`), 사용자(`김승욱`), 검색, 알림, 사용자 메뉴를 고정 배치한다.
- 중앙 페르소나 스위처는 `관리자 콘솔 / 검토자 / 고객사 온보딩` 3개 화면을 전환한다.
- 화면 전환은 별도 페이지 이동이 아니라 동일 앱 내부 상태 변경으로 처리한다. 과제 평가자가 제품 흐름을 끊기지 않고 훑어볼 수 있게 하기 위함이다.
- 관리자 화면의 좌측 사이드바는 `운영 · Operations`와 `거버넌스 · Governance`로 구분한다. 설정형 자동화와 내부통제 기능이 같은 제품 안에 있다는 점을 강조한다.

### 1.2 제품 메시지

| 메시지 | 화면에서 증명하는 방식 |
| --- | --- |
| 온톨로지는 고객사 차이를 코드가 아니라 설정 데이터로 흡수한다 | 온톨로지 스튜디오의 표준 객체, 고객사 Overlay, 매핑 진행률 |
| 자동화는 무조건 자동 승인하지 않고 신뢰도·규정·리스크로 분기한다 | 워크플로우 빌더의 90/95 임계값, 검토자 AI 근거 패널 |
| 회계 자동화는 감사와 권한 통제가 기본 기능이어야 한다 | 권한 매트릭스, SoD 경고, 감사 로그 |
| 온보딩이 제품화되어야 SaaS 마진이 가능하다 | 7단계 위저드, AI 추천 매핑, 연결 테스트, 샘플 테스트 |
| 과금은 처리량과 성과 이벤트에서 나온다 | 대시보드 KPI와 미터링 이벤트 |

### 1.3 데이터와 수치 기준

- 테넌트: `㈜한빛리테일` / 유통·리테일·프랜차이즈 / Tenant `RTL-0042`
- MVP 워크플로우: `법인카드 자동 검토`, `세금계산서 3-Way Matching`
- 신뢰도 임계값: 법인카드 90, 세금계산서 95
- 대시보드 기준 처리량: 월 8,412건, 자동 처리율 42.3%, 예외 탐지 128건, 절감 시간 320h
- 데이터 성격: 모든 회사명, 사용자명, 금액, 계정, 수치는 과제 제출용 가상 데이터

## 3. 관리자 콘솔

관리자 콘솔은 운영 성과, 온톨로지 설정, 워크플로우 정책, 내부통제, 연동 상태, 감사·과금 이벤트를 관리하는 화면이다. 사용자는 회계팀장, CSM/운영자, IT/SAP 관리자, 내부감사 담당자를 가정한다.

### 3.1 대시보드

**목적**  
테넌트의 자동화 성과와 운영 리스크를 한 화면에서 파악한다. PRD의 North Star Metric인 "검토자 개입 없이 정상 완료된 자동 처리 건수"를 운영 지표로 드러낸다.

**핵심 구성**

- KPI 카드: 자동 처리율 42.3%, 예외 탐지 128건, 절감 시간 320h, 총 처리 건수 8,412건
- 일별 처리량 차트: 최근 14일 처리 건수와 자동 처리율 추이를 함께 표시
- 워크플로우 카드: 법인카드 자동 검토와 세금계산서 3-Way의 자동 처리율, 대기 건, 일괄 후보, 월 처리량
- SAP/외부 연동 상태: S/4HANA, 전자세금계산서 API, 카드사 API의 연결·지연 상태
- 신뢰도 분포: 90~100 자동 후보, 70~89 일괄 검토, 0~69 수동 검토 비율
- 최근 감사 이벤트: AI 자동 승인, 임계값 변경, 예외 탐지, SAP posting 등 최신 이력

**기능 동작**

- 워크플로우 카드 클릭 시 해당 워크플로우 빌더 또는 검토자 큐로 이동하는 구조를 전제한다.
- 연동 지연 카드의 `재연결` 또는 `재처리` 액션은 연동 관리 탭의 Retry Queue와 연결된다.
- 자동 처리율과 절감 시간은 미터링 이벤트(`processed_item`, `auto_approved`, `time_saved_min`)에서 집계되는 값으로 정의한다.

### 3.2 온톨로지 스튜디오

**목적**  
표준 Finance Ontology와 고객사별 계정·조직·규정 차이를 매핑한다. "고객사 차이를 코드 수정이 아니라 설정 데이터로 흡수한다"는 제품 차별점을 보여준다.

**핵심 구성**

- 객체 타입 목록: 계정과목, 전표, 증빙, 거래처, 세무항목, 비용센터, 조직·역할, 워크플로우, 규정
- 그래프 캔버스: 전표를 중심으로 계정과목, 증빙, 거래처, 비용센터, 워크플로우, 규정의 관계를 시각화
- 표준/Overlay 범례: 표준 온톨로지 객체와 고객사 Overlay 객체를 구분
- 선택 객체 상세: 객체 수, 관계 수, 표준↔고객사 매핑 현황, 속성, 관계, 사용 중인 워크플로우
- 매핑 진행률: 전체 매핑 89%, 계정과목 218/240 매핑, 미매핑 22건

**기능 동작**

- 좌측 객체를 클릭하면 그래프의 관련 노드와 우측 상세 패널이 함께 변경된다.
- `객체 추가`는 표준 객체를 확장하는 액션으로, 실제 제품에서는 권한이 있는 온톨로지 운영자에게만 노출한다.
- `매핑 편집`은 고객사 계정과목, SAP 코드, 규정 항목을 표준 객체에 연결하는 작업으로 연결된다.
- 미매핑 항목은 온보딩 Step 3의 계정과목 매핑 큐와 같은 백로그로 간주한다.

**주요 객체 예시**

| 객체 | 주요 속성 | 대표 관계 | 기능적 의미 |
| --- | --- | --- | --- |
| 계정과목 | 계정코드, 계정명, 계정유형, 부가세 구분 | 전표 라인에 계상, 세무항목과 연결 | AI 추천 계정과목과 자동 승인 조건의 기준 |
| 전표 | 전표번호, 전표일자, 전표유형, 금액, 승인자 | 증빙 첨부, 거래처 귀속, 비용센터 배부 | SAP posting과 감사 로그의 중심 객체 |
| 거래처 | 사업자번호, 거래처명, 업종, 세금계산서 유형 | 세금계산서 발행, 전표 귀속 | 3-Way Matching과 중복/위험 거래처 판단 기준 |
| 규정 | 문서명, 버전, 적용 범위, 임베딩 상태 | 워크플로우 참조, 계정 규칙 정의 | RAG 판단 근거와 정책 검토의 원천 |

### 3.3 워크플로우 빌더

**목적**  
법인카드와 세금계산서 자동화 흐름을 설정형 파이프라인으로 편집한다. 자동 승인, 일괄 승인 후보, 수동 검토의 분기 기준을 사용자가 이해하고 조정할 수 있게 한다.

**핵심 구성**

- 워크플로우 전환: `법인카드 검토`, `세금계산서 3-Way`
- 단계 파이프라인: 수집 → 정규화 → 증빙/3-Way 매칭 → 규정 검토 → AI 판단
- 분기 결과: 자동 승인/자동 매칭, 일괄 승인 후보, 수동 검토의 예상 비율
- 임계값 슬라이더: 법인카드 90, 세금계산서 95
- 자동 승인 조건 체크리스트: 모든 조건을 만족해야 자동 처리 후보가 됨
- 저장/활성 토글: 운영 반영 전 변경 내용을 검토하는 구조

**기능 동작**

- 워크플로우 탭 전환 시 파이프라인 단계, 임계값, 자동 승인 조건 문구가 변경된다.
- 임계값 슬라이더를 낮추면 자동 처리 예상 비율은 올라가고 수동 검토 비율은 내려간다. 반대로 임계값을 높이면 오승인 위험을 줄이는 대신 자동 처리율이 낮아진다.
- 법인카드는 `자동 승인`, 세금계산서는 `자동 매칭`이라는 용어를 사용한다. 금액·구매오더·입고 리스크가 큰 세금계산서는 95점 이상을 기본값으로 둔다.
- 운영 초기에는 실제 자동 승인보다 `일괄 승인 후보` 중심으로 시작하고, 검토자 수정률을 본 뒤 임계값을 조정한다.

**자동 승인 조건 예시**

| 워크플로우 | 자동 처리 조건 |
| --- | --- |
| 법인카드 | 증빙 존재, 금액 정책 한도 내, 거래처·업종 허용, 과거 사례·계정과목 일치, AI 신뢰도 임계값 이상, 최근 30일 내 반려 이력 없음 |
| 세금계산서 | 공급자 사업자번호 일치, 공급가액·세액·합계 일치, 구매오더·입고 데이터 존재, 수량·단가 허용 오차 내, 중복 세금계산서 아님, AI 신뢰도 임계값 이상 |

### 3.4 규정 센터

**목적**  
사내 규정, 세법, 구매·세무 절차 문서를 Knowledge Layer에 등록하고 워크플로우 판단 근거로 연결한다.

**핵심 구성**

- 문서 테이블: 문서명, 유형, 버전, 적용 워크플로우, 임베딩 상태, 수정일
- 등록 문서: 법인카드 사용·비용처리 규정, 접대비 처리 지침, 부가가치세 매입세액 공제 기준, 거래처 등록·세금계산서 검수 규정 등
- 임베딩 상태: 검색 가능, 임베딩 중
- 적용 워크플로우: 법인카드, 세금계산서 3-Way, 공통
- 문서 업로드 버튼

**기능 동작**

- 문서 업로드 후 파싱, 조항 인식, 임베딩, 검색 가능 상태로 전환되는 처리 흐름을 전제한다.
- 검색 가능 문서만 검토자 AI 근거 패널의 참조 규정으로 노출한다.
- 오래된 문서 또는 임베딩 실패 문서는 규정 품질 리스크로 표시하고, 온보딩 또는 관리자 알림으로 재확인을 요구한다.

### 3.5 권한 관리

**목적**  
역할, 조직, 데이터 범위, 승인 권한을 하나의 매트릭스로 관리하고 직무분리(SoD) 위험을 점검한다.

**핵심 구성**

- SoD 통제 배너: 위반 0건, 경고 1건
- 조직도: 재무회계본부, 경영관리본부, IT본부, 감사실
- 역할 목록: CFO, 회계팀장, 회계담당자, IT/SAP 관리자, 내부감사
- 권한 컬럼: 데이터 범위, 조회, 승인, 수정, 자동승인 설정, 온톨로지/워크플로우 편집, 권한 관리
- 권한 상태: 전체 허용, 조건부 허용, 권한 없음

**기능 동작**

- 역할별 권한 셀을 클릭하면 실제 제품에서는 권한 범위 편집 모달을 연다.
- 데이터 범위는 전사, 본부, 팀, 마스킹 조회 등으로 구분한다.
- 회계팀장이 자동승인 설정과 전표 승인 권한을 동시에 보유하면 SoD 경고를 발생시킨다.
- 내부감사는 조회와 감사 로그 접근은 가능하지만 승인·수정·정책 편집은 제한한다.

### 3.6 연동 관리

**목적**  
SAP, 홈택스, 카드사, 문서 저장소 등 외부 시스템의 연결 상태, 동기화 로그, 오류 재처리 큐를 관리한다.

**핵심 구성**

- 연동 카드: S/4HANA OData, 전자세금계산서 API, 신한·삼성·현대 카드사 API
- 상태 표시: 연결됨, 지연, 오류 건수, 마지막 동기화 시각
- Retry Queue: 현대카드 매입내역 타임아웃, SAP 전표 posting 잠금 충돌, 홈택스 인증 만료
- Sync Log: SAP 구매오더 수집, 홈택스 수취, 카드사 동기화 실패, SAP posting 완료 등
- 전체 동기화, 재연결, 재처리 버튼

**기능 동작**

- `재처리` 클릭 시 해당 오류 건을 다시 큐에 넣고, 성공/실패 결과를 감사 로그에 남긴다.
- 연동 지연 상태는 대시보드의 운영 리스크 카드와 연결된다.
- MVP에서는 SAP 읽기 연동을 우선하고, SAP posting은 제한된 범위의 운영 이벤트로 표현한다.

### 3.7 감사 로그

**목적**  
AI 자동 처리, 검토자 승인·반려, 정책 변경, SAP posting, 권한 변경 이력을 감사 대응 가능한 형태로 보존한다.

**핵심 구성**

- 필터: 전체, AI, 사용자, 시스템, 액션 유형, 날짜
- 로그 테이블: 시각, 행위자, 액션, 대상, 근거, 신뢰도, 결과
- 행위자 구분: AI Engine, 사용자, SAP Connector
- 대표 이력: AI 자동 승인 42건, 임계값 90→92 변경, 세금계산서 예외 탐지, 증빙 누락 반려
- CSV 내보내기 버튼

**기능 동작**

- 모든 자동 처리와 수동 처리는 append-only 로그로 남긴다.
- AI 판단의 근거 규정, 신뢰도, 처리 결과를 함께 기록해 사후 감사와 이의제기에 대응한다.
- CSV 내보내기는 내부감사 또는 월말 리포트 산출용 기능으로 정의한다.

### 3.8 미터링

**목적**  
처리량, 자동 처리, 예외 탐지, SAP 반영, 절감 시간을 과금 이벤트로 기록하고 예상 과금을 산출한다.

**핵심 구성**

- 이벤트 카드: `processed_item`, `auto_approved`, `exception_detected`, `sap_posted`, `time_saved_min`
- 일별 과금 이벤트 추이 차트
- 이번 달 예상 과금: Pro 플랜 기본료, 포함 처리량, 성과 과금, SAP posting 포함 여부
- 예상 합계: 15,534,150원
- 과금 기준일: 2026년 6월

**기능 동작**

- `processed_item`은 사용량 과금의 분모다.
- `auto_approved`는 자동화 성과 과금의 핵심 이벤트다.
- `reviewer_touched`와 `reviewer_overridden`은 자동 처리율과 품질 지표 산정에 사용한다.
- 성과 과금은 고객 신뢰를 위해 구독료의 20~30% 상한을 두는 구조로 표현한다.

## 4. 검토자 화면

검토자 화면은 AI가 자동 처리하지 못한 건을 사람이 빠르게 판단하는 업무 화면이다. 좌측 큐, 중앙 상세, 우측 판단 근거의 3단 구조를 유지한다.

### 4.1 공통 구조

**핵심 구성**

- 상단 워크플로우 탭: 법인카드 검토 37건, 세금계산서 3-Way 24건
- 필터: 리스크 전체, 부서, 금액, 마감 임박
- 일괄 처리 토글: 동일 근거 묶음 처리 모드 전환
- 좌측 큐: 업무 건 목록, 리스크 태그, 금액, 담당자/거래처, 신뢰도
- 중앙 상세: 선택 건의 원천 데이터, 추천 결과, 증빙/비교 테이블
- 우측 AI 판단 근거: 신뢰도 게이지, 점수 구성요인, 판단 문장, 참조 규정, 유사 사례
- 하단 액션: 보류, 반려, 수정 승인, 승인

**기능 동작**

- 좌측 큐에서 건을 선택하면 중앙 상세와 우측 근거 패널이 함께 갱신된다.
- 키보드 단축키 `A`는 승인, `R`은 반려로 동작한다. 입력 필드 포커스 중에는 단축키를 무시한다.
- 승인/반려/보류/수정 승인 후에는 다음 큐 항목으로 자동 이동한다.
- 모든 액션은 감사 로그와 검토자 수정 피드백으로 기록되는 것을 전제한다.

### 4.2 법인카드 검토

**목적**  
자동 처리 임계값 90에 미달한 법인카드 거래를 리스크순으로 검토해 승인, 반려, 보류, 수정 승인한다.

**대표 케이스**

- `신라호텔 다이닝` 480,000원, 김민수, 영업1팀, 2026-06-13 23:47
- 리스크 태그: 심야 사용, 증빙 누락, 고액
- AI 추천 계정과목: 813 기업업무추진비
- 대체 후보: 811 복리후생비, 812 여비교통비
- 신뢰도: 48, 자동 임계값 90 대비 42점 미달

**판단 근거**

- 점수 구성: 규정 부합 38, 증빙 일치 5, 과거 사례 유사도 62, 데이터 정합성 88
- 참조 규정: 접대비 처리 지침 제6조 2항
- 판단 문장: 심야 시간대 고액 사용이며 증빙이 누락되어 개인 사용 가능성을 배제할 수 없음
- 유사 사례: 유사 심야 F&B 반려, 접대비 증빙 보완 후 수정 승인

**기능 동작**

- `수정 승인`은 AI 추천 계정과목 또는 증빙 상태를 검토자가 바꾼 뒤 승인하는 액션이다.
- 증빙 누락 건은 자동 승인 불가이며, 보류 또는 반려를 권장한다.
- 낮은 신뢰도 건은 일괄 처리 묶음에서 제외한다.

### 4.3 세금계산서 3-Way Matching

**목적**  
세금계산서, 홈택스, SAP 구매·입고 데이터를 비교해 금액·수량·품목·사업자번호 불일치를 검토한다.

**대표 케이스**

- `(주)대성물류`, 세금계산서 `#2026-0612`, 13,200,000원
- 리스크 태그: 수량 불일치, 입고 미검수
- 신뢰도: 61, 자동 임계값 95 대비 34점 미달
- 비교 항목: 사업자번호, 공급가액, 세액, 합계, 품목, 수량, 단가

**판단 근거**

- 수량: 세금계산서 1,200, 홈택스 1,200, SAP 입고 1,140
- 차이: 60개, 분할 납품 또는 미검수 가능성
- 점수 구성: 사업자·거래처 일치 100, 금액 일치 60, 수량·입고 일치 45, 단가 정합성 95
- 참조 규정: 거래처·세금계산서 검수 규정 제5조 3항

**기능 동작**

- 불일치 행은 시각적으로 강조하고, 테이블 하단 경고 배너에 원인 후보를 제시한다.
- 정상 건은 `자동 매칭 후보`, 미검수 건은 `보류`, 중복 의심 또는 PO 미매칭 건은 `거래처 문의` 또는 반려를 유도한다.
- 세금계산서는 금액 리스크가 높으므로 법인카드보다 높은 임계값 95를 사용한다.

### 4.4 일괄 처리 UX

**목적**  
같은 근거와 낮은 리스크를 가진 반복 건을 한 번에 처리해 검토자 처리량을 높인다. 단, 서로 다른 판단 근거를 무리하게 묶어 승인하지 않도록 제한한다.

**기능 동작**

- `일괄 처리` 토글을 켜면 좌측 큐에 체크박스가 나타난다.
- AI 추천 묶음 카드가 노출되고, `묶음 선택`을 누르면 저위험·신뢰도 75점 이상 건이 자동 선택된다.
- 1건 이상 선택 시 하단 일괄 처리 바가 표시된다.
- 하단 바는 선택 건 수와 "동일 근거 묶음 일괄 처리" 문구를 표시한다.
- `일괄 승인`, `일괄 보류` 액션은 선택 건을 처리하고 선택 상태를 초기화한다.
- 실제 제품에서는 동일 규정, 동일 계정과목, 동일 리스크 수준, 동일 처리 권한을 만족할 때만 일괄 승인을 허용한다.

## 5. 고객사 온보딩

온보딩 화면은 신규 고객이 제품 사용 전 필요한 설정을 CSM 지원 아래 완료하는 셀프서비스 위저드다. 목표는 개발 작업 없이 연결, 매핑, 권한, 규정, 정책, 샘플 테스트를 끝내는 것이다.

### 5.1 공통 구조

**핵심 구성**

- 좌측 진행 트래커: 7단계, 현재 단계, 완료/진행/대기 상태, 진행률
- 우측 단계 콘텐츠: 입력 폼, 선택 카드, 테이블, 업로드 영역, 테스트 결과

**기능 동작**

| 고객 계정 | 표준 객체 | 신뢰도 | 상태 |
| --- | --- | ---: | --- |
| 81300 접대비 | 813 기업업무추진비 | 98% | 자동 |
| 81100 복리후생 | 811 복리후생비 | 96% | 자동 |
| 82600 도서구입 | 826 도서인쇄비 | 89% | 검토 |
| 90100 잡비 | 매핑 필요 | 54% | 미매핑 |

**기능 동작**

- 신뢰도 90% 이상은 자동 매핑 후보로 표시한다.
- 70~89%는 검토 상태로 두고 사용자가 후보를 확인한다.
- 낮은 신뢰도 또는 표준 객체가 없는 항목은 미매핑 상태로 남긴다.
- `추천 일괄 승인`은 자동 매핑 후보를 한 번에 확정하는 액션이다.
- 확정된 매핑은 온톨로지 스튜디오의 매핑 진행률에 반영된다.

### 5.5 Step 4 · 조직 · 권한

**목적**  
SAP HR 또는 고객 조직 정보를 가져와 역할과 데이터 범위를 설정한다.

**핵심 구성**

- 조직도 가져오기: 27개 노드, 재무회계본부, 경영관리본부, IT본부, 감사실
- SAP HR 연동 완료 배지
- 역할 매핑: CFO, 회계팀장, 회계담당자, IT/SAP 관리자, 내부감사
- 데이터 범위 자동 설정
- SoD 규칙 적용 표시

**기능 동작**

- 조직 정보 가져오기 후 기본 역할 템플릿을 자동 추천한다.
- 권한 범위는 이후 관리자 콘솔의 권한 관리 매트릭스와 같은 모델을 사용한다.
- 전표 생성과 승인, 자동승인 설정과 최종 승인처럼 충돌 가능한 권한은 SoD 검증 대상이다.

### 5.6 Step 5 · 규정 문서

**목적**  
AI 판단 근거로 사용할 사내 회계 규정 문서를 업로드하고 검색 가능 상태를 확인한다.

**핵심 구성**

- 드래그앤드롭 업로드 영역
- 지원 형식: PDF, DOCX, HWP
- 제한: 최대 50MB
- 문서 상태 집계: 4개 검색 가능, 2개 임베딩 중
- 문서 목록: 법인카드 사용·비용처리 규정 v3.2, 접대비 처리 지침 v2.1, 여비교통비 지급 규정 v1.7 등

**기능 동작**

- 업로드된 문서는 파싱, 조항 인식, 임베딩 단계를 거친다.
- 임베딩 중 문서는 진행률을 표시한다.
- 검색 가능 문서는 검토자 화면의 참조 규정과 워크플로우 정책 검토에 사용된다.
- 오래된 문서 또는 누락 문서는 CSM 확인 또는 고객 재업로드 대상으로 분류한다.

### 5.7 Step 6 · 워크플로우 · 정책

**목적**  
운영할 워크플로우와 초기 자동 처리 정책을 선택한다.

**핵심 구성**

- 워크플로우 선택: 법인카드 자동 검토, 세금계산서 3-Way Matching
- 임계값: 법인카드 90, 세금계산서 95
- 초기 정책: 자동 승인보다 일괄 승인 후보 중심
- PoC 안내: 초기 4~6주 동안 성과 측정과 검토자 피드백 수집

**기능 동작**

- 고객은 운영할 워크플로우를 선택한다.
- 임계값과 자동 승인 조건은 관리자 콘솔의 워크플로우 빌더 기본값으로 이어진다.
- 초기에는 보수적 설정으로 시작하고, 검토자 수정률과 자동 처리 후보율을 보고 조정한다.

### 5.8 Step 7 · 샘플 테스트

**목적**  
운영 전 샘플 데이터로 예상 자동 처리율과 주요 예외를 확인한다.

**핵심 구성**

- 샘플 데이터: 최근 법인카드 100건, 세금계산서 60건
- 테스트 실행 버튼
- 예상 자동 처리율, 검토 필요 건, 예외 유형 결과 영역
- 운영 시작 버튼

**기능 동작**

- `테스트 실행` 클릭 시 테스트 중 상태를 거쳐 결과를 표시한다.
- 결과는 PoC 성공기준, 임계값 조정, 운영 시작 여부 판단에 사용한다.
- `운영 시작`은 온보딩 완료 이벤트를 발생시키고 관리자 대시보드 운영 상태로 연결된다.

## 10. 향후 실제 제품화 시 보강 항목

- 일괄 처리 안전장치: 동일 규정·동일 계정·동일 리스크 검증 실패 시 일괄 승인 비활성화

- 권한 승인 플로우: SoD 경고 해소 요청, 승인자 지정, 변경 이력 감사 로그
- 미터링 명세: 고객에게 제공할 월별 과금 상세, 제외 건, 수동 수정 건, 성과 과금 상한 표시
