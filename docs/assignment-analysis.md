## 1. 한눈에 보는 과제 개요

이 과제의 핵심 미션은 기존에 구축형(SI) 방식으로 납품하던 **온톨로지 기반 재무·회계 AI 자동화 솔루션**을, 반복 판매 가능한 **범용 제품(Product)** 으로 전환하는 전략과 기획을 제안하는 것이다.

회사가 보고 싶은 것은 단순 기능 아이디어가 아니라 아래 역량이다.

- 기존 기술 자산을 **제품 자산·마진 구조**로 해석하는 능력
- 타겟 시장과 수익 모델을 설계하는 사업 감각 (특히 "가장 많은 이익" 관점)
- 온톨로지 기반 접근의 차별성을 **제품 경제성**으로 연결하는 관점
- 자동화 워크플로우와 Human Review UX를 구체적으로 설계하는 능력
- 구축형과 SaaS형을 모두 고려한, 숫자가 있는 가격 전략

## 2. 회사가 이미 보유한 자산

반드시 그대로 쓸 필요는 없지만, 제품화 전략의 강력한 출발점이다.

| 영역 | 보유 자산 |
| --- | --- |
| Finance Ontology 코어 | 계정과목·전표·증빙·거래처·세무항목 등 회계 개념·관계 온톨로지 |
| 권한 모델 | 역할·조직·데이터 범위 기반 권한 모델 |
| 워크플로우 정의 | 코드 수정 없이 온톨로지 편집으로 자동화 업무 추가·변경 |
| 지식 기반 RAG | 회계규정·세법·과거 분개 패턴 RAG (벡터 검색 + 온톨로지 그래프 결합) |
| 테넌시 관리 | 고객사별 문서 관리 및 임베딩 |
| LLM 라우터 | 민감 데이터는 사내 LLM, 일반 업무는 외부 API로 정책 기반 라우팅 |
| 자동화 워크플로우 | 법인카드 검토, 3-Way Matching, 마감 체크리스트, 계정잔액 이상탐지, 전표 자동분개 |
| ERP 연동 | SAP Connector (ECC BAPI / S/4HANA OData / 커스텀 RFC) |
| 프론트엔드 | React 관리자 콘솔, 검토자용 Review UI |

**자산 해석의 핵심:** 위 목록은 "기능"이 아니라 **"고객사 차이를 흡수하는 표준화 레이어"** 로 읽어야 한다. SI가 매번 다시 짜던 것(규정·계정과목·권한·SAP 환경)을 온톨로지+어댑터+RAG가 **설정으로 흡수**한다는 것이 제품화의 전제다. 이 지점이 §3의 마진 논리로 직결된다.

## 3. 핵심 논지: 온톨로지가 곧 마진 엔진이다 (★ 가장 중요)

과제의 최종 질문은 "**가장 많은 이익을 낼 수 있는 관점**"이다. 여기에 답하려면 온톨로지의 강점을 "설명 가능성" 같은 기술 자랑이 아니라 **원가 구조**로 번역해야 한다.

### 3.1 SI vs Product의 경제학

| 항목 | 구축형(SI) | 온톨로지 기반 Product |
| --- | --- | --- |
| 매출 인식 | 프로젝트 단발 | 구독(반복) |
| 온보딩 원가 | 고객당 수개월 인력 투입 | 고객당 수일 설정(셀프서비스+CSM) |
| 매출↔인력 관계 | 선형(고객↑ = 인력↑) | 준선형(고객↑ ≫ 인력↑) |
| 매출총이익률 | 30~45% (인건비 지배) | 70~80% 지향 |
| 지식 재사용 | 프로젝트 종료 시 사장 | 온톨로지·규정 템플릿으로 축적·재판매 |

**온톨로지가 없으면** 고객마다 규정·계정과목·권한·SAP 매핑을 코드로 새로 짜야 하고, 이는 온보딩 원가를 고객당 수개월로 고정시킨다 = SI 마진 구조에 갇힌다.

**온톨로지가 있으면** 고객 차이가 "코드"가 아니라 "설정 데이터"로 흡수된다. 온보딩이 개발이 아니라 **매핑 작업**이 되고, 반복될수록 표준 온톨로지·산업별 템플릿이 두꺼워져 온보딩 원가가 계속 내려간다. 즉 **온톨로지 = 한계원가를 낮추는 자산**이며, 이것이 "가장 많은 이익"의 직접적 근거다.

### 3.2 제품화가 살려야 할 온톨로지 강점 6가지

1. **설정형 확장**: 새 워크플로우/규정을 코드 배포 없이 온톨로지·정책 편집으로 추가 → 신규 자동화의 한계원가↓, 출시 속도↑.
2. **차이 흡수**: 고객사별 계정과목·규정·조직·SAP 환경을 매핑/어댑터로 흡수 → 온보딩 표준화.
3. **설명 가능한 판단**: AI 결론을 LLM 확률이 아니라 규정·과거사례·객체 관계 그래프로 근거화 → 회계·감사 도메인 필수 요건(감사 추적) 충족, 규제 방어.
4. **권한·데이터 범위 내장**: 자동화와 권한 모델이 한 그래프에 있어 내부통제/직무분리(SoD)를 자동으로 만족 → 엔터프라이즈 진입 장벽 통과.
5. **재사용 자산 축적**: 산업별 온톨로지 템플릿·규정 팩이 쌓일수록 신규 고객 온보딩이 빨라지는 복리 효과(네트워크성 원가 우위).
6. **멀티ERP 이식성**: 판단 로직이 SAP에 종속되지 않고 온톨로지에 있어, ERP 어댑터만 갈아끼우면 시장 확장 가능(향후 TAM 확대).

> 한 줄 메시지: **"온톨로지는 기능이 아니라, SI를 SaaS 마진으로 바꾸는 원가 자산이다."**

### 3.3 단, 마진은 자동으로 오지 않는다 (조건부 논지)

위 70~80% 마진은 "목표"이지 "기본값"이 아니다. 온톨로지 강점이 가장 큰 고객(규정·권한·SAP가 복잡한 대기업)일수록 온보딩이 가장 무겁다 — **차별성이 극대화되는 조건 = 제품화가 가장 어려운 조건**이라는 긴장이 존재한다. 이 긴장을 통제하지 못하면 매핑·규정 세팅이 매 고객 커스텀 작업으로 번져 그대로 SI 원가로 회귀한다.

따라서 마진의 실제 결정 변수는 "온톨로지가 있느냐"가 아니라 다음 셋이다.

1. **시장 선택** — 복잡도를 통제할 수 있는 고객부터 시작(§6: 중견기업 + BPO/SSC 우선, 대기업 Phase 2).
2. **온보딩 제품화** — 표준 온톨로지 선탑재 + 추천 매핑 + 산업 템플릿으로 초기 세팅을 사람이 아니라 제품이 처리(§11.3).
3. **SI화 방어 규율** — 고객 요청마다 코드를 여는 대신 설정·정책·템플릿으로 흡수(§12).

정리하면 **온톨로지는 마진의 필요조건, 시장 선택·온보딩 제품화가 충분조건**이다. 이 셋이 맞물릴 때만 SI 마진(30~45%)이 SaaS 마진(70~80%)으로 실제 전환된다.

## 4. 제품 정의 및 포지셔닝

### 4.1 제품 정의

단순 회계 자동화 도구가 아니라 다음으로 정의한다.

> **고객사별 회계 규정·권한·ERP 환경·검토 프로세스 차이를 온톨로지로 흡수하는, 설명 가능한 Finance AI Automation Platform**

이 정의는 "기존 SI 자산을 반복 판매 가능한 제품으로 전환한다"는 메시지를 선명하게 만든다.

### 4.2 경쟁 포지셔닝

| 경쟁군 | 한계 | 본 제품의 차별점 |
| --- | --- | --- |
| RPA (UiPath 등) | 화면 자동화 중심, 규정 변경·예외에 취약 | 회계 객체·규칙을 온톨로지로 모델링 → 정책 변경에 강함 |
| OCR/문서처리 | 추출은 되나 회계 판단까지 어려움 | 증빙·전표·거래처·세무항목·규정 연결로 판단 근거 제공 |
| ERP Add-on | 특정 ERP에 종속 | SAP 어댑터+온톨로지 매핑으로 고객 차이 흡수, 멀티ERP 이식성 |
| 일반 LLM 자동화 | 설명가능성·권한·감사추적 취약 | RAG+그래프+권한모델 기반 신뢰 가능한 자동화 |
| 국내 지출/세무 SaaS (스펜딧·비즈플레이·더존 등) | 단일 업무(지출/세금계산서)에 특화, 회계 판단·통합 자동화는 약함 | 회계 도메인 전반을 온톨로지로 묶어 워크플로우 확장·통합 |
| 컨설팅/SI | 고객별 구축 비용·기간 큼 | 표준 워크플로우+설정형 온보딩으로 반복 판매 |

**포지셔닝 한 줄:** "RPA는 화면을 흉내 내고, LLM은 답을 추측한다. 우리는 **회계를 이해하는 자동화**다."

## 5. 제품 비전 및 전략 스토리

- **Vision:** 회계팀의 반복 검토 업무를 "AI가 처리하고 사람이 확인하는" 구조로 전환하되, 모든 판단이 규정과 근거로 설명되는 신뢰 가능한 자동화.
- **Wedge(진입점):** 처리량이 크고 ROI 증명이 쉬운 단일 워크플로우로 랜딩 → 회계팀 전체 마감 자동화로 확장(land & expand).
- **Moat(해자):** 축적되는 산업별 온톨로지·규정 템플릿 + 고객 데이터로 학습되는 추천 정확도 + SAP/내부통제 통합 깊이.

## 6. 타겟 시장 및 ICP (선정 이유 포함)

### 6.1 왜 이 시장인가 (Why now)

- 국내 중견·대기업 다수가 SAP(ECC→S/4HANA 전환기)를 사용 → **표준 커넥터 하나로 다수 고객 커버** 가능.
- 회계팀 반복 업무(법인카드·세금계산서·마감)는 노동집약적이고 오류·감사 리스크가 크다 → 자동화 ROI가 명확.
- 전자세금계산서·홈택스·내부회계관리제도(내부통제) 등 **규제가 데이터·감사추적을 강제** → 설명 가능한 온톨로지 접근이 규제 적합성에서 유리.

### 6.2 세그먼트 우선순위 (Tiering)

| 티어 | 대상 | 선정 근거 | 진입 우선순위 |
| --- | --- | --- | --- |
| **Beachhead** | SAP/표준 ERP 쓰는 **중견기업** (제조·유통·프랜차이즈·커머스) | 처리량↑=ROI 명확 + 대기업보다 의사결정 빠름·세일즈 사이클 짧음 | 즉시 |
| **Beachhead** | **회계 BPO·기장/세무 대행·아웃소싱** | 1개 계약으로 다수 고객사 처리 → 온톨로지 재사용 효과 극대화(이 제품에 가장 이상적) | 즉시 |
| **Tier 1.5** | 그룹사 Shared Service Center | 계열사 반복 업무·표준화 니즈 큼 | Phase 1.5 |
| **Phase 2** | SAP **대기업** | ACV 크고 내부통제 니즈 강하나 6~18개월 세일즈·보안/법무/PoC 장벽 | 레퍼런스 확보 후 |

### 6.3 ICP (Ideal Customer Profile) 정의 — Beachhead

- **중견기업:** 매출 수백억~수천억, SAP 또는 표준 ERP, 월 법인카드·세금계산서 수천~수만 건, 소수 회계 인력에 검토 부담이 집중.
- **회계 BPO/SSC:** 다수 고객사·계열사 회계를 반복 처리 → 온톨로지·템플릿 재사용 효과가 가장 큰 이상적 고객.
- 트리거: 회계 인력 채용난, 마감 단축(fast close) 압박, 인력 증원 없이 처리량 증가를 감당해야 하는 상황.
- 구매 의사결정: 중견=CFO/회계팀장 직접, BPO=대표/운영총괄 → 대기업 대비 **짧은 결재선·빠른 PoC**.

> 초점: **"절감이 숫자로 보이고 의사결정이 빠른 고객"** 부터 잡는다. 대기업은 여기서 쌓은 레퍼런스·템플릿을 진입권 삼아 Phase 2로 공략.

## 7. 핵심 기능: MVP 워크플로우 2개 상세

두 워크플로우를 MVP로 선정한 이유: 처리량↑(ROI 증명 쉬움), 예외 빈발(자동화 가치↑), 온톨로지 연결을 보여주기 좋음(증빙·거래처·규정·SAP), 검토 UX 설계 적합, KPI 측정 용이.

### 7.1 워크플로우 1 — 법인카드 내역 자동 검토

**흐름**
1. 법인카드 승인 내역 수집(카드사/SAP)
2. 증빙·사용 내역 매칭
3. 사내 규정·부서·직급·비용유형 기준 정책 검토(RAG+온톨로지)
4. 계정과목·세무항목 추천
5. AI 판단 근거·신뢰도 산출
6. 신뢰도 임계값 이상 → 자동 승인/자동 전표 생성
7. 예외 건 → 검토자 화면 전달

**검토 UX**
- 대기 목록 리스크 점수순 정렬 / 위반 가능 규정 하이라이트
- 과거 유사 처리 사례 표시 / 추천 계정과목 및 대체 후보
- 승인·반려·수정승인·**일괄 승인(동일 근거 묶음)** 제공

**KPI:** 자동 처리율, 건당 검토 시간, 정책 위반 탐지율, 전표 자동화율, 검토자 수정률

### 7.2 워크플로우 2 — 세금계산서 3-Way Matching

**흐름**
1. 거래처 발행 세금계산서 수집
2. 홈택스 데이터 수집
3. SAP 구매오더·입고·검수 데이터와 매칭
4. 금액·수량·공급가액·세액·거래처·품목 불일치 탐지
5. 불일치 원인 후보 제시
6. 신뢰도 높은 정상 건 자동 승인
7. 예외 건 검토자 전달

**검토 UX**
- 세금계산서·홈택스·SAP를 한 화면 비교 / 불일치 필드 하이라이트
- AI 판단 원인·참조 데이터 표시 / 거래처별 반복 이슈 패턴
- 일괄 승인·보류·거래처 문의·SAP 수정요청 액션

**KPI:** 자동 매칭률, 예외 처리 시간, 오매칭률, 월 마감 지연 감소율, 거래처 문의 감소율

## 11. 화면 설계 방향

### 11.1 관리자 콘솔 — "설정 가능한 제품"
- **Ontology Studio**: 회계 객체·관계·속성 편집(+버전/변경이력)

- **Document Knowledge**: 규정 문서 업로드·임베딩 상태·참조 관리
- **Integration**: SAP·홈택스·카드사·기타 커넥터
- **Monitoring**: 자동 처리율·예외율·절감시간·오류율

### 11.2 검토자 화면 — "회계 담당자의 작업대"
- 좌: 검토 대기 목록 / 상단: 필터·우선순위·업무유형·마감임박
- 중앙: 대상 건 상세 / 우: AI 판단 근거·참조 규정·유사 사례·신뢰도
- 하단: 승인·수정승인·반려·보류·일괄 처리

UX 원칙: ①결론보다 **근거를 먼저** 신뢰 가능하게 ②신뢰도는 숫자만이 아니라 **근거 구성요소**로 분해 ③일괄처리는 **동일 근거 묶음** 중심 ④검토자 수정 → 온톨로지·추천 모델 개선으로 환류(learning loop)

### 11.3 고객사 온보딩 — "제품화의 심장"
1. 회사 기본정보 → 2. SAP 연결 테스트 → 3. 계정과목표 가져오기 → 4. 표준 온톨로지↔고객 계정 매핑 → 5. 조직·권한 설정 → 6. 규정 문서 업로드 → 7. 워크플로우 선택 → 8. 신뢰도 임계값·자동승인 정책 → 9. 샘플 데이터 테스트 → 10. 운영 전환

> 온보딩 소요일이 곧 마진(§3). "사람이 몇 달 하던 세팅"을 "CSM 지원 하의 수일 셀프서비스"로 압축하는 것이 제품화의 성패.

## 13. 가격 전략 (단가·미터링 포함)

> ⚠️ 아래 금액은 국내 B2B SaaS 관행 기반 **가정치**다(§15). **가격 정본은 PRD §15**이며, 본 절은 요약·논리용으로 PRD 수치에 맞춘다.

### 13.1 구축형 (대기업·보안 강요 고객 — Phase 2)
- 진단/PoC: **8,000만~1.5억 원** (8~12주, 1개 워크플로우)
- 초기 구축비: **3억~8억 원** (SAP 연동·온톨로지 매핑·권한·규정 세팅)
- 연간 라이선스: **1.2억~3억 원** / 유지보수: 라이선스 또는 구축비의 **18~22%**
- 워크플로우 추가: **5,000만~1.5억 원/건**, 전용 LLM·프라이빗 배포·보안 옵션 별도

### 13.2 SaaS형 (중견·BPO/SSC·표준 ERP)
| 플랜 | 월 기본료(가정) | 포함 범위 | 초과 과금 |
| --- | --- | --- | --- |
| Starter | 500만 원 | 워크플로우 1개, 월 5천 건, 기본 Review UI | 처리 건당 50~150원~ |
| Pro | 1,500만 원 | 워크플로우 2개, 월 3만 건, SAP 연동, KPI | 건당 + 애드온 |
| BPO/SSC | 2,000만 원 + 테넌트당 과금 | 다수 고객사·계열사, 공통 템플릿 + 고객사별 Overlay | 테넌트·처리량 |
| Enterprise | 3,000만 원+ | 워크플로우 3개+, 월 10만 건+, 고급 권한·감사 | 처리량 + 성과 보너스 |

미터링 대상: 전표·세금계산서·법인카드 **건수**, 검토자 수, 워크플로우 수, 테넌트 수, 고급 옵션.

### 13.3 성과 기반 과금 (상세)

| 지표 | 미터링 방식 | 과금 방식 |
| --- | --- | --- |
| 자동 처리 건수 | 검토자 개입 없이 완료된 건수 | 건당 과금(구독 초과분) |
| 절감 시간 | (도입 전 평균 처리시간 − 도입 후) × 건수 | 절감시간 환산액의 일정 %를 성과 보너스 |
| 자동 승인율 | 전체 처리 중 자동 승인 비율 | 목표 초과 시 인센티브 |
| 예외 탐지 건수 | 정책위반·증빙누락·금액불일치 탐지 | 탐지 건당 또는 리포트 패키지 |
| 전표 생성 건수 | SAP 자동 생성/제안 전표 수 | 건당 과금 |

**추천 구조: 기본 구독료 + 처리량 과금 + (엔터프라이즈) 성과 보너스.**
- "절감액의 몇 %"만으로 가면 고객 저항·산정 분쟁이 크다.
- 예측 가능한 기본료로 고객 안심 + 자동화가 커질수록 공급사 수익↑(인센티브 정렬).
- 성과 보너스는 **도입 전 베이스라인 합의**를 전제로만 청구(분쟁 예방).

### 13.4 경쟁력 있는 가격 포인트 근거 (ROI 논리)

- 회계 담당자 1인 인건비 ≈ 연 6,000만 원(가정). Beachhead 고객이 자동화로 **1~2 FTE** 검토 시간을 절감하면 연 절감 ≈ **6,000만~1.2억 원**.
- Starter(연 6,000만) 단일 워크플로우로도 ROI 설명 가능. Pro(월 1,500만=연 1.8억)는 처리량 큰 중견 또는 BPO/SSC가 다수 고객사·계열사를 묶을 때 정당화.
- 즉 가격은 "절감 가치의 일부만 청구"하는 **value-based pricing** 으로 정당화 → 고객은 남는 장사, 공급사는 높은 마진(§3).

- 초기 고객은 SAP/표준 ERP를 쓰는 중견기업·회계 BPO·그룹사 SSC이며, 대기업은 레퍼런스 확보 후 Phase 2다.
- 고객사는 내부 회계 규정 문서를 보유하고 RAG로 업로드 가능하다.
- 자동 승인은 초기엔 저위험 업무에 한정, 고위험은 Human Review를 거친다.
- 성과 측정을 위해 도입 전/후 처리시간 베이스라인을 합의·비교할 수 있다.
- **§13의 모든 금액은 국내 B2B SaaS 관행 기반 가정치이며 실제 원가·경쟁가로 보정 필요.**
- 회계 담당자 인건비, FTE 절감 규모는 예시 가정이다.
