## 1. Executive Summary

### 1.1 제품 한 줄 정의

**Finance Ontology AI Automation Platform은 고객사별 회계 규정, 권한, ERP 환경, 검토 프로세스 차이를 온톨로지로 흡수해 재무·회계 반복 업무를 안전하게 자동화하는 엔터프라이즈 AI 운영 플랫폼이다.**

### 1.2 핵심 관점

이 제품은 단순한 회계 자동화 도구가 아니다. 기존 RPA, OCR, AP 자동화 솔루션이 특정 업무를 자동화하는 데 초점을 둔다면, 본 제품은 기업마다 다른 **회계 개념, 데이터 관계, 승인 권한, 내부 규정, ERP 구조**를 온톨로지로 모델링하고 이를 기반으로 자동화 워크플로우를 확장한다.

제품화의 핵심은 기존 구축형 프로젝트에서 반복되던 커스터마이징 요소를 **설정 가능한 제품 레이어**로 바꾸는 것이다.

- 고객사별 계정과목, 전표, 증빙, 거래처, 세무항목 매핑
- 고객사별 조직, 역할, 데이터 접근 권한
- 고객사별 사내 회계 규정과 예외 처리 정책
- 고객사별 SAP 환경 차이
- 업무별 자동 승인 기준과 Human Review 기준

이를 통해 고객은 코드 개발 없이 자동화 범위를 확장하고, 공급자는 SI성 구축 매출을 반복 가능한 SaaS/플랫폼 매출로 전환할 수 있다.

### 1.3 추천 MVP

MVP는 다음 두 워크플로우를 중심으로 출시한다.

1. **법인카드 내역 자동 검토**
2. **세금계산서 3-Way Matching**

두 업무는 처리량이 많고, 회계 규정·증빙·거래처·SAP 데이터 간 관계가 중요하며, 자동화 성과를 자동 처리율, 절감 시간, 예외 탐지율로 측정하기 쉽다.

### 1.4 핵심 전략 논지: 온톨로지 = 마진 엔진

과제의 최종 질문은 "가장 많은 이익을 낼 수 있는 관점"이다. 이 제품이 이익을 극대화하는 근본 이유는 온톨로지가 **SI 사업을 SaaS 마진 구조로 전환하는 원가 자산**이기 때문이다.

- 온톨로지가 없으면 고객마다 계정과목·규정·권한·SAP 매핑을 코드로 새로 짜야 하고, 온보딩 원가가 고객당 수개월로 고정된다(= SI 마진 30~45%, 인건비 지배).
- 온톨로지가 있으면 고객 차이가 "코드"가 아니라 "설정 데이터"로 흡수되어, 온보딩이 개발이 아니라 매핑 작업이 된다. 반복될수록 표준 온톨로지·산업 템플릿이 두꺼워져 한계원가가 계속 내려간다(= SaaS 마진 70~80% 지향).

즉 온톨로지는 "설명 가능성"이라는 기능이기 이전에, **고객이 늘어도 인력이 그만큼 늘지 않는 구조**를 만드는 이익 엔진이다. 상세는 §3.4 참조.

다만 이 마진은 자동으로 실현되지 않는다. 온톨로지의 강점이 가장 큰 고객(규정·권한·SAP가 복잡한 대기업)일수록 온보딩도 무거워, 통제하지 못하면 그대로 SI 원가로 되돌아간다. 그래서 온톨로지는 마진의 *필요조건*일 뿐이고, **시장 선택과 온보딩 제품화가 충분조건**이다. 이 PRD가 초기 타겟을 대기업이 아니라 중견기업·BPO로 잡는 근거가 여기에 있다(§3.4, §4).

## 2. 문제 정의

### 2.1 고객 문제

중견기업과 대량 회계 처리 조직은 반복 업무를 자동화하고 싶지만, 실제 운영에서는 다음 이유로 자동화가 제한된다.

- 계정과목, 비용 정책, 승인 권한이 회사마다 다르다.
- SAP는 고객사별 커스터마이징이 많아 표준 연동만으로는 부족하다.
- 법인카드, 세금계산서, 구매오더, 입고, 전표, 증빙이 서로 다른 시스템에 흩어져 있다.
- AI가 결론을 제안해도 회계 담당자는 판단 근거, 참조 규정, 감사 추적 없이는 신뢰하기 어렵다.
- 고위험 건은 자동 처리보다 검토자 개입이 필요하다.
- 자동화 성과를 비용 절감이나 처리량 기준으로 측정하기 어렵다.

### 2.2 기존 접근의 한계

| 접근 | 강점 | 한계 |
| --- | --- | --- |
| RPA | 화면 조작 자동화가 빠름 | 화면 변경, 예외 상황, 규정 변경에 취약 |
| OCR | 문서 데이터 추출에 강함 | 회계적 판단, 권한, 내부 규정 반영이 약함 |
| 일반 LLM | 자연어 처리와 요약에 강함 | 설명 가능성, 권한 통제, 감사 추적, 데이터 정합성에 약함 |
| ERP Add-on | ERP 내부 프로세스와 밀접 | 고객사별 확장성과 비ERP 데이터 결합에 한계 |
| SI 구축 | 고객 맞춤 구현 가능 | 반복 판매가 어렵고 구축 기간·비용이 큼 |

### 2.3 제품 기회

회계 자동화 시장은 이미 AP 자동화, 비용 관리, ERP 연동 중심으로 형성되어 있다. 그러나 엔터프라이즈 고객은 단순 문서 처리보다 **복잡한 내부 규정, 권한, SAP 커스터마이징, 감사 대응**을 함께 해결해야 한다.

따라서 본 제품의 기회는 "AI로 회계 업무를 자동화한다"가 아니라, **고객사별 회계 운영 모델을 온톨로지로 구성하고 그 위에서 AI 자동화를 안전하게 실행한다**에 있다.

## 3. 제품 비전과 포지셔닝

### 3.1 제품 비전

회계팀이 반복 검토 업무에 시간을 쓰는 대신, 예외 판단과 재무적 의사결정에 집중할 수 있도록 재무·회계 업무의 운영 지식을 제품화한다.

### 3.2 포지셔닝 문장

**SAP를 사용하는 중견기업과 회계 BPO/SSC를 위한 온톨로지 기반 Finance AI Automation Platform.**

본 제품은 법인카드, 세금계산서, 전표, 증빙, 거래처, 계정과목, 세무항목, 권한, 내부 규정을 하나의 의미 모델로 연결하고, AI 판단 근거와 신뢰도를 제공하여 자동 처리와 Human Review를 안전하게 분기한다.

### 3.3 차별점

| 차별점 | 설명 |
| --- | --- |
| 온톨로지 기반 회계 모델 | 계정과목, 전표, 증빙, 거래처, 세무항목, 권한, 워크플로우를 연결된 도메인 모델로 관리 |
| 설정형 워크플로우 | 자동화 업무를 코드 수정이 아니라 온톨로지 편집과 정책 설정으로 확장 |
| 설명 가능한 AI | AI 판단 결과와 함께 참조 규정, 과거 사례, 데이터 관계, 신뢰도 근거를 노출 |
| Human-in-the-loop | 신뢰도 임계값에 따라 자동 처리와 검토자 처리 분기 |
| SAP 어댑터 전략 | ECC BAPI, S/4HANA OData, 커스텀 RFC 등 고객사별 SAP 차이를 어댑터로 흡수 |
| 성과 기반 과금 가능성 | 처리 건수, 자동 처리율, 절감 시간, 예외 탐지 건수를 제품 로그로 측정 |

### 3.4 온톨로지 기반 접근의 제품화 경제성 (핵심 논지)

온톨로지의 강점은 기술 자랑이 아니라 **원가 구조**로 번역할 때 사업적 의미가 생긴다. 이 절이 "가장 많은 이익" 요구에 대한 본 제품의 직접적 답이다.

#### SI vs Product 경제학

| 항목 | 구축형(SI) | 온톨로지 기반 Product |
| --- | --- | --- |
| 매출 인식 | 프로젝트 단발 | 구독(반복) |
| 온보딩 원가 | 고객당 수개월 인력 투입 | 고객당 수일 설정(셀프서비스 + CSM) |
| 매출 ↔ 인력 관계 | 선형 (고객↑ = 인력↑) | 준선형 (고객↑ ≫ 인력↑) |
| 매출총이익률 | 30~45% (인건비 지배) | 70~80% 지향 |
| 지식 재사용 | 프로젝트 종료 시 사장 | 온톨로지·규정 템플릿으로 축적·재판매 |

핵심은 온보딩 원가다. 온톨로지가 고객 차이를 "설정 데이터"로 흡수하는 순간, 온보딩이 개발이 아니라 매핑 작업이 되고, 고객이 늘수록 표준 온톨로지·산업 템플릿이 두꺼워져 **한계원가가 복리로 하락**한다. 이것이 반복 판매 가능한 SaaS 마진을 만든다.

#### 제품화가 살려야 할 온톨로지 강점 6가지

1. **설정형 확장** — 새 워크플로우·규정을 코드 배포 없이 온톨로지·정책 편집으로 추가 → 신규 자동화 한계원가↓, 출시 속도↑.
2. **차이 흡수** — 고객사별 계정과목·규정·조직·SAP 환경을 매핑/어댑터로 흡수 → 온보딩 표준화.
3. **설명 가능한 판단** — AI 결론을 LLM 확률이 아니라 규정·과거사례·객체 관계 그래프로 근거화 → 감사 추적 등 회계 도메인 필수 요건 충족.
4. **권한·데이터 범위 내장** — 자동화와 권한 모델이 한 그래프에 있어 내부통제·직무분리(SoD)를 자동 충족 → 엔터프라이즈 진입 장벽 통과.
5. **재사용 자산 축적** — 산업별 온톨로지 템플릿·규정 팩이 쌓일수록 신규 온보딩이 빨라지는 복리 효과(네트워크성 원가 우위).
6. **멀티ERP 이식성** — 판단 로직이 SAP에 종속되지 않고 온톨로지에 있어, 어댑터만 교체하면 시장 확장 가능(향후 TAM 확대).

> 한 줄 메시지: **"온톨로지는 기능이 아니라, SI를 SaaS 마진으로 바꾸는 원가 자산이다."**

#### 단, 이 마진은 자동으로 오지 않는다 (조건부 논지)

위 표의 70~80% 마진은 "목표"이지 "기본값"이 아니다. 온톨로지의 강점이 가장 크게 드러나는 고객(규정·권한·SAP가 복잡한 대기업)은 동시에 온보딩이 가장 무거운 고객이다. 즉 **차별성이 극대화되는 조건 = 제품화가 가장 어려운 조건**이라는 긴장이 존재한다. 이 긴장을 통제하지 못하면 온톨로지 매핑·규정 세팅이 매 고객마다 사람에 의존하는 커스텀 작업으로 번지고, 앞서 벗어나려던 SI 원가로 그대로 회귀한다.

따라서 마진의 실제 결정 변수는 "온톨로지가 있느냐"가 아니라 다음 세 가지다.

1. **시장 선택** — 복잡도를 통제할 수 있는 고객부터 시작한다. (§4: 중견기업 + BPO/SSC 우선, 대기업은 Phase 2)
2. **온보딩 제품화** — 표준 온톨로지 선탑재 + 추천 매핑 + 산업 템플릿으로 초기 세팅을 사람이 아니라 제품이 처리하게 만든다.
3. **SI화 방어 규율** — 고객 요청마다 코드를 여는 대신 설정·정책·템플릿으로 흡수한다. (§16: 제품화보다 SI화 리스크)

정리하면 **온톨로지는 마진 엔진의 필요조건이고, 시장 선택과 온보딩 제품화가 충분조건**이다. 이 셋이 맞물릴 때만 SI 마진(30~45%)이 SaaS 마진(70~80%)으로 실제 전환된다.

## 4. 시장 전략

### 4.1 시장 선택 원칙

이 제품은 대기업으로 갈수록 ROI가 커지지만, 동시에 제품화 난이도와 현금 회수 리스크도 커진다. 따라서 초기 시장은 "가장 큰 고객"이 아니라 **온톨로지의 재사용 효과를 빠르게 증명할 수 있고, 세일즈 사이클이 짧으며, SI화 위험을 통제할 수 있는 고객**으로 잡는다.

시장 선택 기준:

- 월 회계 처리량이 충분히 많아 자동화 ROI가 명확할 것
- 고객사별 규정·권한·계정과목 차이가 있어 온톨로지 차별성이 드러날 것
- 단일 워크플로우 PoC로 4~6주 안에 가치 검증이 가능할 것
- SAP 또는 표준 ERP 연동 범위를 읽기 중심으로 시작할 수 있을 것
- 고객이 너무 보수적이지 않아 AI 보조 검토를 실제 운영에 태워볼 수 있을 것
- 한 번 만든 온톨로지·규정·워크플로우 템플릿을 다른 고객에게 재사용할 수 있을 것

### 4.2 1차 타겟 ICP

초기 타겟은 **SAP 사용 중견기업 + 회계 BPO/SSC**로 설정한다. 대기업은 높은 ACV가 매력적이지만, 초기 제품이 바로 들어가기에는 세일즈 사이클, 보안 심사, PoC 관문, 커스터마이징 부담이 크다. 따라서 대기업은 레퍼런스와 템플릿이 쌓인 뒤 Phase 2에서 공략한다.

| 타겟 | 우선순위 | 선정 이유 | 주요 리스크 | 대응 |
| --- | --- | --- | --- | --- |
| SAP 사용 중견기업 | Beachhead | 처리량과 ROI가 충분하고 대기업보다 의사결정이 빠름 | SAP 커스터마이징, 온보딩 부담 | 단일 워크플로우 PoC, 읽기 연동 우선, 표준 매핑 템플릿 |
| 회계 BPO / 기장·세무 대행 / 회계 아웃소싱 | Beachhead | 1개 고객이 다수 고객사의 회계 업무를 처리해 온톨로지 재사용 효과가 큼 | 고객사별 편차, 데이터 품질 | 멀티테넌트 템플릿, 고객사별 Overlay, 처리량 기반 과금 |
| 그룹사 SSC | Phase 1.5 | 계열사별 반복 업무가 많고 표준화 니즈가 강함 | 내부 이해관계자와 권한 복잡도 | 한 계열사 또는 한 업무로 시작 후 확장 |
| SAP 대기업 | Phase 2 | ACV가 크고 내부통제·감사 니즈가 강함 | 6~18개월 세일즈 사이클, 보안·법무·PoC 장벽 | 중견/BPO 레퍼런스 확보 후 진입, 구축형 패키지로 분리 |

1차 업종은 다음 순서로 공략한다.

| 우선순위 | 섹터 | 선정 이유 |
| --- | --- | --- |
| 1 | 유통·리테일·프랜차이즈 | 매장·지점·거래처·법인카드 사용량이 많고 예외 검토 부담이 큼 |
| 2 | 제조 중견기업 | 구매오더, 입고, 세금계산서, SAP 데이터가 연결되어 3-Way Matching 가치가 큼 |
| 3 | 플랫폼·커머스 | 거래처와 정산 데이터가 많고 빠른 마감이 중요 |
| 4 | 회계 BPO·SSC | 다수 고객사 또는 계열사에 같은 워크플로우를 반복 적용할 수 있음 |

### 4.3 왜 이 시장인가: 장점과 리스크를 함께 본다

SAP 기반 중견·대기업 시장은 겉으로는 매력적이지만, 그 매력은 곧 리스크이기도 하다. 따라서 초기 전략은 대기업 정면 돌파가 아니라 **중견+BPO에서 반복 가능한 제품 패턴을 먼저 증명하고, 대기업은 검증된 패키지로 확장**하는 것이다.

| 장점으로 보이는 요소 | 뒤집으면 생기는 리스크 | 제품 전략 |
| --- | --- | --- |
| 처리량이 많아 ROI가 크다 | 이미 SAP 자체 기능, RPA, SI, 컨설팅사가 진입한 경쟁 시장이다 | 대기업 정면 경쟁보다 법인카드·세금계산서 단일 워크플로우로 빠른 PoC |
| 규정·권한이 복잡해 온톨로지가 빛난다 | 온보딩이 지옥이 되고 SI로 회귀할 수 있다 | 표준 Finance Ontology + 고객사 Overlay + 산업별 템플릿으로 설정화 |
| SAP 커넥터가 진입장벽이다 | 우리에게도 개발·유지보수 부담이 큰 장벽이다 | MVP는 읽기 연동·파일 업로드·제한된 posting부터 시작하고 어댑터를 점진 확대 |
| 높은 ACV를 기대할 수 있다 | 세일즈 사이클이 길고 현금 회수가 느리다 | 중견/BPO에서 4~6주 PoC와 월 구독/사용량 과금으로 빠른 매출 검증 |
| 감사·내부통제가 구매 이유다 | AI 오류에 대한 허용도가 낮고 도입 의사결정이 보수적이다 | 자동 승인보다 Human Review와 일괄 승인 후보로 시작, 감사 로그와 근거 노출을 기본 제공 |

이 접근의 핵심은 "대기업을 포기한다"가 아니다. 대기업이 요구하는 수준의 제품 안정성은 최종적으로 필요하지만, 처음부터 대기업의 복잡도를 모두 떠안으면 제품이 아니라 SI가 된다. 초기에는 중견기업과 BPO에서 반복 가능한 온톨로지·워크플로우·온보딩 패턴을 축적하고, 이를 대기업 진입권으로 사용한다.

### 4.4 시장 근거

시장 근거는 정량 추정치보다 방향성 중심으로 사용한다.

- SAP는 Finance 영역에서 AI 기반 프로세스가 예측 정확도, 보고 주기 단축, 의사결정, 리스크 및 컴플라이언스 관리 개선에 기여한다고 설명한다.
- 국내 전자세금계산서는 법인사업자에게 의무화되어 있고, 개인사업자도 매출 기준에 따라 의무 발급 대상이 확대되어 왔다. 이는 세금계산서 데이터가 디지털 형태로 축적되는 기반이 된다.
- 연결 내부회계관리제도는 자산 규모별로 순차 적용되어 상장사와 그룹사의 내부통제·감사 대응 부담을 키운다.
- 글로벌 AP 자동화 제품들은 invoice capture, PO matching, approval routing, ERP sync, audit trail을 핵심 가치로 제시한다. 이 영역은 이미 수요가 검증된 시장이다.

### 4.5 초기 GTM

초기 Go-To-Market은 대기업 장기 세일즈보다 **고빈도 단일 워크플로우 PoC → 유료 운영 → 워크플로우 확장**이 적합하다.

1. SAP 사용 중견기업 또는 회계 BPO를 대상으로 1개 워크플로우 PoC를 제안한다.
2. 법인카드 검토 또는 세금계산서 3-Way Matching 중 고객의 처리량이 가장 큰 업무로 시작한다.
3. 초기 PoC는 4~6주로 제한하고, SAP write-back 없이 읽기 연동·파일 업로드·샘플 데이터로 가치를 먼저 검증한다.
4. 자동 처리 후보율, 검토 시간 절감, 예외 탐지 유효율, 검토자 수정률을 측정한다.
5. PoC 성공 시 월 구독 + 처리량 과금으로 유료 운영 전환한다.
6. 같은 고객 안에서 두 번째 워크플로우 또는 다수 고객사/계열사로 확장한다.
7. 반복 설정은 표준 온톨로지와 산업별 템플릿으로 흡수해 대기업 Phase 2 진입 자산으로 만든다.

## 5. 사용자와 페르소나

### 5.1 CFO

주요 관심사:

- 비용 절감
- 마감 리드타임 단축
- 내부통제 강화
- 회계 인력 생산성 향상

성공 기준:

- 자동화로 절감된 시간과 비용이 명확히 측정됨
- 감사 리스크와 오류가 감소함
- 추가 인력 없이 처리량 증가를 감당함

### 5.2 회계팀장 / Controller

주요 관심사:

- 반복 검토 업무 감소
- 예외 건 우선순위화
- 업무 표준화
- 규정 변경 반영

성공 기준:

- 검토자는 예외 건에만 집중함
- AI 판단 근거가 회계 실무자가 납득할 수준으로 제공됨
- 규정 변경 시 운영자가 직접 정책을 수정할 수 있음

### 5.3 회계 담당자 / Reviewer

주요 관심사:

- 오늘 처리해야 할 검토 건을 빠르게 파악
- AI 추천의 근거 확인
- 일괄 처리
- 수정 결과가 다음 추천에 반영됨

성공 기준:

- 검토 화면에서 필요한 근거를 한 번에 확인
- 반복적인 정상 건은 자동 또는 일괄 승인
- 오류 건은 빠르게 반려 또는 보류 처리

### 5.4 IT / SAP 관리자

주요 관심사:

- SAP 연동 안정성
- 권한과 보안
- 운영 장애 최소화
- 커스터마이징 부담 감소

성공 기준:

- SAP 연결 상태와 동기화 로그를 확인 가능
- 장애 발생 시 재처리와 원인 추적 가능
- 고객사별 SAP 차이를 설정과 어댑터로 흡수

### 5.5 내부감사 / 컴플라이언스 담당자

주요 관심사:

- 의사결정 로그
- 권한 통제
- 정책 위반 추적
- 감사 대응 증빙

성공 기준:

- 누가, 언제, 어떤 근거로 승인했는지 추적 가능
- AI 자동 승인 건도 판단 근거와 신뢰도 로그가 남음
- 정책 변경 이력과 영향 범위를 확인 가능

## 6. 제품 구조

### 6.1 핵심 구성

| 구성 요소 | 설명 |
| --- | --- |
| Finance Ontology Core | 회계 객체, 관계, 속성, 권한, 워크플로우를 정의하는 도메인 모델 |
| Tenant Ontology Overlay | 고객사별 계정과목, 조직, 규정, 승인 권한, SAP 매핑을 표준 온톨로지 위에 확장 |
| Knowledge Layer | 사내 규정, 세법, 과거 처리 사례, 분개 패턴을 RAG와 그래프로 결합 |
| Workflow Engine | 업무별 자동화 단계, 조건, 예외 분기, 신뢰도 임계값 실행 |
| AI Decision Service | 분류, 매칭, 이상 탐지, 분개 추천, 판단 근거 생성 |
| Human Review UI | 예외 검토, 근거 확인, 일괄 처리, 수정 피드백 |
| Integration Layer | SAP, 홈택스, 카드사, 문서 저장소, ERP 연동 |
| Audit & Metering Layer | 의사결정 로그, 권한 로그, 처리량, 자동화 성과, 과금 이벤트 기록 |

### 6.2 온톨로지 기반 제품화 원칙

제품화 단계에서 온톨로지는 내부 기술이 아니라 고객 가치로 노출되어야 한다.

원칙:

- 표준 Finance Ontology를 기본 제공한다.
- 고객사는 자신의 계정과목, 조직, 권한, 규정을 표준 모델에 매핑한다.
- 워크플로우는 회계 객체와 관계를 기반으로 구성된다.
- AI 판단은 참조한 객체, 규정, 과거 사례와 함께 설명된다.
- 고객사의 수정·승인 패턴은 온톨로지와 지식 기반 개선에 반영된다.

### 6.3 데이터 흐름

1. 외부 시스템에서 원천 데이터 수집
2. 표준 Finance Ontology 객체로 정규화
3. 고객사별 Tenant Overlay와 매핑
4. RAG/그래프 기반 관련 규정과 과거 사례 검색
5. Workflow Engine이 업무별 판단 단계 실행
6. AI Decision Service가 추천 결과, 신뢰도, 근거 생성
7. 신뢰도 임계값에 따라 자동 처리 또는 Human Review 분기
8. 처리 결과를 SAP에 반영
9. 모든 이벤트를 Audit & Metering Layer에 기록

## 7. MVP 범위

### 7.1 MVP 목표

MVP 목표는 1~2개 고객사에서 다음을 증명하는 것이다.

- 고객사별 회계 규정과 SAP 매핑을 설정형으로 흡수할 수 있다.
- 두 가지 고빈도 회계 워크플로우에서 자동 처리율과 절감 시간을 측정할 수 있다.
- 검토자가 AI 판단 근거와 신뢰도를 보고 실제 업무 처리에 사용할 수 있다.
- 제품 로그를 기반으로 사용량/성과 기반 과금이 가능하다.

### 7.2 MVP 포함 기능

- 법인카드 내역 자동 검토
- 세금계산서 3-Way Matching
- 관리자 콘솔 기본형
- 검토자 Review UI
- 고객사 온보딩 기본형
- SAP 연동 패키지
- 사내 규정 문서 업로드 및 RAG 검색
- 신뢰도 임계값 기반 자동/수동 분기
- 처리 이벤트 로그와 KPI 대시보드

### 7.3 MVP 제외 기능

- 완전한 노코드 워크플로우 빌더
- 모든 ERP 지원
- 전표 자동분개 전면 자동화
- 산업별 온톨로지 마켓플레이스
- 고객사 간 벤치마크 리포트
- 완전 셀프서브 가입과 결제

## 8. 핵심 워크플로우 1: 법인카드 내역 자동 검토

### 8.1 목적

법인카드 사용 내역을 사내 규정, 증빙, 조직, 비용 유형, 과거 처리 패턴과 비교하여 정상 건은 자동 처리하고, 리스크가 있는 건은 검토자에게 전달한다.

### 8.2 입력 데이터

- 카드 승인 내역
- 카드 매입 내역
- 영수증·증빙 이미지 또는 전자 증빙
- 사용자, 부서, 직급, 비용센터
- 사내 비용 규정
- 계정과목표
- 과거 유사 처리 사례
- SAP 전표 또는 비용 처리 데이터

### 8.3 주요 판단 항목

- 증빙 누락 여부
- 사용 가능 시간·장소·업종 위반 여부
- 한도 초과 여부
- 주말·심야·휴일 사용 여부
- 개인 사용 가능성
- 중복 청구 가능성
- 적정 계정과목
- 부가세 공제 가능 여부
- 승인권자 필요 여부

### 8.4 기본 흐름

1. 카드 승인 내역과 증빙을 수집한다.
2. 거래처, 업종, 금액, 시간, 사용자, 부서를 표준 객체로 변환한다.
3. 고객사 비용 규정과 조직 권한을 조회한다.
4. 과거 유사 처리 사례와 분개 패턴을 검색한다.
5. 정책 위반 가능성과 적정 계정과목을 판단한다.
6. AI 판단 근거와 신뢰도 점수를 생성한다.
7. 신뢰도와 리스크 수준에 따라 자동 승인, 일괄 승인 후보, 수동 검토로 분기한다.
8. 승인 건은 SAP 전표 생성 또는 전표 생성 후보로 전달한다.
9. 검토자의 수정 결과를 학습 피드백으로 기록한다.

### 8.6 검토자 UX

검토자 화면은 다음 정보를 제공한다.

- 리스크 점수순 검토 대기 목록
- 정책 위반 가능성 태그
- AI 추천 계정과목과 대체 후보
- 참조된 사내 규정 문구
- 과거 유사 처리 사례
- 신뢰도 점수와 점수 구성 요인
- 증빙 원본 미리보기
- 승인, 수정 승인, 반려, 보류, 일괄 처리 액션

일괄 처리는 "동일 규정, 동일 계정과목, 동일 리스크 수준"으로 묶인 건에 한정한다.

### 8.7 자동 처리 기준

초기 MVP에서는 보수적으로 설정한다.

- 증빙이 존재한다.
- 금액이 정책 한도 내에 있다.
- 거래처와 업종이 허용 범위에 있다.
- 과거 유사 처리와 계정과목 추천이 일치한다.
- AI 신뢰도가 90점 이상이다.
- 최근 30일 내 동일 조건의 검토자 반려 이력이 없다.

조건을 모두 만족하면 자동 승인 후보가 된다. 운영 초기에는 자동 승인 대신 일괄 승인 후보로 먼저 제공하고, 고객이 임계값을 조정할 수 있게 한다.

### 8.8 KPI

- 실제 자동 승인율
- 검토자 수정률
- 건당 평균 검토 시간
- 정책 위반 탐지 건수

- SAP 전표 생성 자동화율

## 9. 핵심 워크플로우 2: 세금계산서 3-Way Matching

### 9.1 목적

거래처 발행 세금계산서, 홈택스 데이터, SAP 구매·입고 데이터를 대조하여 정상 건은 자동 매칭하고, 금액·수량·거래처·품목 불일치 건은 검토자에게 전달한다.

### 9.2 입력 데이터

- 거래처 발행 세금계산서
- 홈택스 전자세금계산서 수취 데이터
- SAP 구매오더
- SAP 입고·검수 데이터
- 거래처 마스터
- 품목 마스터
- 계약·단가 정보
- 과거 매칭 이력

### 9.3 주요 판단 항목

- 공급자 사업자번호 일치 여부
- 공급가액·세액·합계금액 일치 여부
- 구매오더와 품목·수량 일치 여부
- 입고·검수 완료 여부
- 중복 세금계산서 여부
- 단가 차이 허용 범위 여부
- 분할 납품 또는 부분 청구 여부
- 거래처별 반복 오류 패턴

### 9.4 기본 흐름

1. 세금계산서와 홈택스 데이터를 수집한다.
2. SAP 구매오더, 입고, 검수 데이터를 조회한다.
3. 거래처, 품목, 금액, 수량을 표준 객체로 정규화한다.
4. 세금계산서, 홈택스, SAP 데이터를 3-Way로 대조한다.
5. 불일치 항목과 원인 후보를 생성한다.
6. 신뢰도 높은 정상 건은 자동 매칭한다.
7. 예외 건은 검토자 화면으로 전달한다.
8. 검토자는 승인, 보류, 거래처 문의, SAP 수정 요청을 수행한다.
9. 처리 결과를 SAP에 반영하고 감사 로그를 남긴다.

### 9.5 상태 정의

| 상태 | 설명 |
| --- | --- |
| Invoice Imported | 세금계산서 수집 완료 |
| Hometax Matched | 홈택스 데이터 매칭 완료 |
| SAP Data Linked | SAP 구매·입고 데이터 연결 완료 |
| Matching Completed | 3-Way Matching 완료 |
| Auto Matched | 자동 매칭 완료 |
| Exception Found | 불일치 발견 |

| Vendor Inquiry | 거래처 문의 필요 |
| SAP Correction Requested | SAP 데이터 수정 요청 |
| Posted / Closed | 처리 완료 |

## 13. 성공 지표

**North Star Metric:** **"검토자 개입 없이 정상 완료된 자동 처리 건수(Auto-completed transactions)"**. 이 지표는 제품 가치(검토 시간 절감)와 매출(처리량·성과 과금)이 동시에 붙는 단일 지표로, 아래 제품·사업·운영 KPI는 모두 이 값을 끌어올리는 방향으로 정렬된다. 자동 처리 건수↑ = 고객 절감↑ = 미터링 매출↑ = 마진↑(§3.4)의 선순환이 핵심이다.

### 13.1 제품 KPI

| KPI | 정의 | MVP 목표 |
| --- | --- | --- |
| Time to Value | 계약 후 첫 자동화 성과 확인까지 걸리는 시간 | 8주 이내 |
| 자동 처리율 | 전체 처리 건 중 검토자 개입 없이 처리된 비율 | 법인카드 40% 이상, 세금계산서 30% 이상 |
| 자동 승인 후보율 | 자동 승인 조건을 만족한 건의 비율 | 법인카드 60% 이상 |
| 검토자 수정률 | AI 추천 대비 검토자가 수정한 비율 | 10% 이하 |
| 예외 탐지 정확도 | 검토자가 유효하다고 판단한 예외 탐지 비율 | 85% 이상 |
| 평균 검토 시간 | 수동 검토 건당 평균 처리 시간 | 기존 대비 50% 감소 |
| 근거 노출률 | AI 판단에 참조 규정 또는 데이터 근거가 연결된 비율 | 95% 이상 |

### 13.2 사업 KPI

| KPI | 정의 | 목표 |
| --- | --- | --- |
| PoC 전환율 | PoC 후 유료 계약 전환 비율 | 50% 이상 |
| ARR per Customer | 고객당 연간 반복 매출 | 1억 원 이상 |
| Gross Margin | SaaS/라이선스 매출 총이익률 | 70% 이상 |
| Expansion Revenue | 동일 고객 내 워크플로우·계열사 확장 매출 | 12개월 내 30% 이상 |
| Implementation Ratio | 구축·설정 비용 대비 반복 매출 비율 | 2년 내 반복 매출 우위 |

> **ARR per Customer 1억+는 확장 후 목표**다. Starter 단독(월 500만·연 6,000만, §15)이 아니라 **Starter로 진입 → 12개월 내 Pro 전환·2차 워크플로우·계열사 확장(Expansion Revenue)** 으로 고객당 1억+에 도달하는 land-and-expand 경로를 전제한다.

## 14. 로드맵

### 14.1 리소스 가정

MVP 개발 리소스는 4-3 로드맵과 동일하게 **개발 2 + 기획 2 = 4인 병렬 스쿼드**로 가정한다. 별도 전담 QA·디자이너·인프라 조직 없이, 기획 트랙이 1스프린트 앞서(spec-ahead) 스펙·수용기준·테스트케이스를 공급하고 개발 2트랙이 빌드 처리량의 상한을 형성한다.

| 트랙 | 역할 | 급 | 주요 책임 |
| --- | --- | :---: | --- |
| D1 | 개발 · 백엔드/코어 | 시니어 | 온톨로지·플랫폼 코어, AI 판단·RAG·신뢰도, 미터링·과금 엔진 |

| P-Sr | 기획 · PM/도메인 | 시니어 | 제품 스펙, 온톨로지 도메인 설계, 우선순위, PoC·GTM·가격, 디자인 겸임 |
| P-Jr | 기획 · 스펙/Q·A | 주니어 | 유저스토리·수용기준, 수동/시나리오 Q·A, 미터링·청구 명세 검증 |

## 15. 가격 전략

가격은 구축형과 SaaS형을 분리하되, 공통적으로 **기본료 + 사용량 + 성과 기반 옵션**을 적용한다.

아래 가격은 과제 답안을 위한 가설이다. 실제 가격은 고객 규모, 처리 건수, SAP 커스터마이징 난이도, 보안 요구사항, 자동화 범위에 따라 검증이 필요하다.

### 15.1 구축형 가격

대상:

- 대기업
- 보안·내부망 요구가 큰 고객
- SAP 커스터마이징이 많은 고객
- 프라이빗 클라우드 또는 온프레미스 배포가 필요한 고객

가격 구조:

| 항목 | 가격 가설 | 설명 |
| --- | --- | --- |
| 진단/PoC | 8천만~1.5억 원 | 8~12주, 1개 워크플로우 기준 |
| 초기 구축비 | 3억~8억 원 | SAP 연동, 온톨로지 매핑, 권한, 규정 문서 세팅 |
| 연간 라이선스 | 1.2억~3억 원 | 워크플로우 2개, 기본 사용량 포함 |
| 유지보수 | 라이선스 또는 구축비의 18~22% | 운영 지원, 업데이트, 장애 대응 |
| 워크플로우 추가 | 5천만~1.5억 원 | 업무 복잡도와 연동 범위에 따라 과금 |
| 전용 LLM/보안 옵션 | 별도 견적 | 사내 LLM, 프라이빗 배포, 고급 보안 옵션 |

### 15.2 SaaS형 가격

대상:

- 중견기업
- 회계 BPO, 세무·기장 대행사, 그룹사 SSC
- 표준 SAP 또는 클라우드 ERP를 사용하는 기업
- 빠른 도입과 낮은 초기비를 원하는 고객

가격 구조:

| 플랜 | 월 기본료 가설 | 포함 범위 |
| --- | --- | --- |
| Starter | 월 500만 원 | 1개 워크플로우, 월 5천 건, 기본 Review UI |
| Pro | 월 1,500만 원 | 2개 워크플로우, 월 3만 건, SAP 연동, KPI 대시보드 |
| BPO / SSC | 월 2,000만 원 + 테넌트당 과금 | 다수 고객사/계열사 관리, 공통 템플릿, 고객사별 Overlay |
| Enterprise SaaS | 월 3,000만 원 이상 | 3개 이상 워크플로우, 월 10만 건 이상, 고급 권한, 감사 리포트 |

초과 사용량:

- 일반 처리 건: 건당 50~150원
- AI 판단 근거 생성 포함 건: 건당 100~300원
- SAP posting 또는 reverse sync 포함 건: 건당 100~500원
- 고급 이상 탐지 또는 분개 추천: 건당 300~1,000원

### 15.3 성과 기반 과금

성과 기반 과금은 고객 신뢰를 위해 처음부터 절감액 배분 방식만 적용하지 않는다. MVP 단계에서는 측정 가능한 이벤트 기반 과금을 중심으로 한다.

추천 모델:

1. 기본 구독료
2. 처리량 기반 과금
3. 자동 처리 성과 보너스
4. 엔터프라이즈 고객 대상 절감액 공유 옵션

미터링 이벤트:

| 이벤트 | 정의 | 활용 |
| --- | --- | --- |
| processed_item | 워크플로우가 처리한 전체 건 | 사용량 과금 |
| auto_approved_item | 검토자 개입 없이 승인된 건 | 성과 과금 |
| reviewer_touched_item | 검토자가 확인한 건 | 자동화율 계산 |
| reviewer_overridden_item | 검토자가 AI 추천을 수정한 건 | 품질 지표 |
| exception_detected_item | 정책 위반·불일치가 탐지된 건 | 리스크 탐지 성과 |
| time_saved_minutes | 기준 시간 대비 절감 시간 | ROI 리포트 |
| sap_posted_item | SAP에 반영된 건 | 고급 사용량 과금 |

성과 과금 예시:

| 모델 | 과금 방식 |
| --- | --- |
| 자동 처리 건 기반 | 자동 승인 완료 건당 100~300원 |
| 절감 시간 기반 | 고객과 합의한 시간당 비용 × 절감 시간 × 10~15% |
| 예외 탐지 기반 | 유효 예외 탐지 건당 1,000~5,000원 |
| 마감 단축 보너스 | 월 마감 일정 목표 달성 시 월 정액 보너스 |

성과 기반 과금의 전제:

- 도입 전 기준 처리 시간을 고객과 합의한다.
- 자동 처리와 검토자 처리의 기준을 명확히 정의한다.
- 검토자 수정 또는 반려 건은 성과 과금에서 제외한다.
- 성과 과금은 월 구독료의 20~30%를 상한으로 두어 예측 가능성을 확보한다.

### 15.4 가격 포인트 근거

글로벌 AP 자동화 제품은 월 구독, 사용자 기반 과금, 거래량 기반 과금을 혼합한다. SMB/미드마켓 제품은 낮은 월 기본료로 시작하지만, 엔터프라이즈 고객은 ERP 연동, 감사 통제, 다법인 구조, 글로벌 권한, 처리량에 따라 가격이 올라간다.

본 제품은 단순 AP 자동화가 아니라 회계 운영 모델을 온톨로지로 구성하는 제품이므로, 가격 기준은 저가형 AP SaaS보다 높게 잡아야 한다. 다만 초기에는 대기업 가격을 먼저 밀어붙이기보다 중견기업과 BPO/SSC에서 PoC로 자동 처리율, 절감 시간, 예외 탐지 성과를 증명해 가격 저항을 낮춘다.

**ROI 근거 (value-based pricing):** 회계 담당자 1인 인건비를 연 6,000만 원으로 가정하면(가정치), Beachhead 고객이 자동화로 1~2 FTE 상당의 검토 시간을 절감할 경우 연간 절감 효과는 약 6천만~1.2억 원이다. Starter 플랜(연 6,000만 원)은 단일 워크플로우만으로 ROI를 설명할 수 있고, Pro 플랜(월 1,500만 원 = 연 1.8억 원)은 처리량이 큰 중견기업 또는 BPO/SSC에서 다수 고객사·계열사 업무를 묶을 때 정당화된다. 즉 가격은 "고객이 얻는 절감 가치의 일부만 청구"하는 구조로 정당화되고, 이 구조가 고객에게는 남는 장사이면서 공급사에는 높은 마진(§3.4)을 남긴다. 실제 단가는 고객별 처리량·인건비 수준에 따라 PoC에서 보정한다.

## 17. 명시적 가정

본 PRD는 과제에서 제공된 정보와 공개 자료를 바탕으로 작성했으며, 부족한 정보는 다음과 같이 가정한다.

- 초기 고객은 SAP를 사용하는 중견기업, 회계 BPO, 그룹사 SSC로 가정한다. SAP 대기업은 레퍼런스 확보 이후 Phase 2 타겟으로 둔다.
- 기존 SAP Connector는 ECC BAPI, S/4HANA OData, 커스텀 RFC를 일정 수준 지원한다.
- 고객사는 사내 회계 규정 문서를 보유하고 있으며, 이를 제품에 업로드할 수 있다.
- 법인카드와 세금계산서 데이터는 API, 파일, 배치 중 하나의 방식으로 수집 가능하다.
- 자동 승인은 초기에는 저위험·고신뢰 건에 한정한다.
- 성과 측정을 위해 도입 전 평균 처리 시간과 도입 후 처리 시간을 비교할 수 있다.
- 가격은 실제 고객 인터뷰와 PoC 결과에 따라 조정이 필요하다.
